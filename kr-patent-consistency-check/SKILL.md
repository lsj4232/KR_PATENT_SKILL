# 한국 특허 명세서 정합성 점검

## 무엇을 하는가

작성된 명세서 초안을 입력받아 청구항·도면·본문 3자 정합성과 한국 특허 가이드라인 준수 여부를 정밀 점검하고, 위반 사항에 대한 수정안과 점수표를 제공한다.

**입력**: 명세서 docx 또는 텍스트, (있으면) 도면 파일들, 청구항 원문
**출력**: 점검 보고서 (위반 항목 + 위치 + 수정안 + 점수)

## 점검 항목 마스터 리스트

### A. 가이드라인 표현 점검 (자동화 가능)

| 항목 | 점검 방법 | 위반 시 |
|---|---|---|
| "종래" 단어 0건 | grep "종래" 본문 | 모든 발생 위치 보고 + 우회 표현 제안 |
| "구성되는" 0건 | grep "구성되는" 본문 | 모든 발생 위치 보고 + "이루어지는" 또는 "포함하는" 제안 |
| 청구항 한정 수치의 배경기술 노출 | 청구항에서 수치(N dB, N mV 등) 추출 → 배경기술에서 grep | 노출된 수치 + 정성적 대체 표현 제안 |
| 해결과제의 수단 시사 표현 | "이용하여", "정량적으로", "선택적으로 ~함으로써" 등 패턴 검사 | 위반 위치 + 목적형 재작성안 |
| 해결과제의 효과 시사 표현 | "확보할 수 있도록", "~이 향상되도록" 등 패턴 검사 | 위반 위치 + 문제의식형 재작성안 |
| 추상적 효과 표현 | "성능이 향상된다", "정확도가 높다" 같은 3단 인과 누락 패턴 검사 | 3단 인과로 재구성 제안 |
| 안전성 위해 표현 | "폭발", "유해", "위험" 등 인체/안전 부정 표현 검사 | 약화된 표현 제안 |

### B. 청구항 매핑 점검

| 항목 | 점검 방법 |
|---|---|
| 청구항 N개 모두 해결수단에 등장 | 청구항별 핵심 어휘를 추출하여 해결수단 본문에서 검색 |
| 청구항 N개 모두 실시예에 등장 | 위와 동일하게 실시예 본문에서 검색 |
| 청구항 원문 변경 0건 | 입력 청구항과 명세서의 청구범위 섹션을 diff |

### C. 부호 매핑 점검 (양방향)

| 점검 방향 | 의미 |
|---|---|
| 부호의 설명 → 본문 | 부호 설명표에 등재된 모든 부호가 본문에 최소 1회 등장하는가 |
| 본문 → 부호의 설명 | 본문에 등장하는 모든 (NNN) 형식 부호가 부호 설명표에 등재되어 있는가 |
| 도면 → 명세서 | 도면에 보이는 부호가 명세서 본문/부호 설명에 모두 있는가 |
| 명세서 → 도면 | 명세서가 언급하는 부호가 해당 도면에 실제 그려져 있는가 |

### D. 도면 텍스트 정합성

| 항목 | 점검 방법 |
|---|---|
| 도면 박스 라벨이 명세서 용어와 동일한가 | 도면에서 텍스트 추출 → 명세서 어휘와 대조 |
| 도면 텍스트가 청구항 한정 수치를 노출하는가 | 도면 텍스트에서 수치 추출 → 청구항 한정 수치와 매칭 (★위험) |
| 도면 표기 (LCS, MUX 등 영문 약어)가 명세서에서도 동일하게 정의되는가 | 영문 약어 추출 → 명세서 첫 등장 시 한글 정의 + 약어 병기 확인 |

### E. 효과 항목 점검

| 항목 | 점검 방법 |
|---|---|
| 효과 항목이 3단 인과(구조→메커니즘→이점) 구조인가 | 각 효과 항목을 분해하여 3요소 존재 여부 확인 |
| 해결과제와 효과의 1:1 매핑 | 해결과제 N번에 대응하는 효과 N번이 있는가 |
| 종속항 부가 효과가 별도 기재되어 있는가 | 종속항이 추가하는 효과를 효과 섹션에서 검색 |

### F. 스토리텔링 일관성

| 항목 | 점검 방법 |
|---|---|
| 배경기술의 문제의식이 해결과제의 목적과 매칭 | 배경 3·4단락의 문제 ↔ 해결과제 1번 등 매칭 |
| 해결과제 → 해결수단 → 효과의 일관성 | 같은 스토리 흐름인가, 도중에 다른 발명처럼 보이는가 |
| 섹션 간 자연스러운 브릿지 문장 존재 | "이에", "이러한 구성에 의해" 등 연결 문구 |

### G. 형식 점검

| 항목 | 점검 방법 |
|---|---|
| 발명의 명칭이 권리범위를 부당히 좁히지 않는가 | 청구항 vs 명칭 비교 |
| 도면의 간단한 설명이 모든 도면 커버 | 도 1 ~ 도 N 모두 |
| "보인" / "나타낸" 표현 통일 | 한 명세서 내 혼용 여부 |
| 요약서가 청구항 1 복붙이 아닌가 | 요약과 청구항 1 텍스트 유사도 |
| 대표도 지정 | 대표도 섹션 존재 |

## 작업 순서

### Step 1. 자료 확인

명세서 docx 또는 텍스트, 도면 파일, 청구항 원문 입력 확인. 누락된 자료가 있으면 사용자에게 요청.

### Step 2. 자동 점검 (A섹션 - 가이드라인 표현)

```bash
# 예시: 명세서 텍스트에서 금지어 검색
extract-text spec.docx > spec.txt
grep -n "종래" spec.txt
grep -n "구성되는" spec.txt
```

각 위반 위치를 줄 번호와 함께 보고. 우회 표현 제안.

### Step 3. 청구항 매핑 점검 (B섹션)

청구항 N개 각각에서 핵심 어휘 2-3개를 추출하여 본문에서 grep. 누락 청구항이 있으면 경고.

### Step 4. 부호 매핑 점검 (C섹션)

- 부호의 설명 섹션 파싱 → 모든 부호 추출 (예: 100, 110, 120, 121, …)
- 본문에서 정규식 `\((\d+)\)` 또는 `\(S\d+\)` 등으로 부호 추출
- 양방향 비교

### Step 5. 도면 점검 (D섹션 - 도면이 있을 때만)

도면이 png/tif/pptx 등으로 제공되면:
- 이미지에서 텍스트 추출 (수동 또는 OCR)
- 명세서 어휘와 대조
- 청구항 한정 수치가 도면 박스 라벨에 노출되어 있는지 점검 (★위험)

### Step 6. 효과/스토리/형식 점검 (E, F, G섹션)

본문을 정독하면서 점검. 자동화 어려움. LLM 자체 판단으로 수행.

### Step 7. 점수표 작성

10개 영역에 대한 가중 점수표:

```
| # | 항목 | 가중치 | 점수 | 가중점 | 평가 |
|---|---|---|---|---|---|
| 1 | 발명의 명칭 | 5% | N/10 | | |
| 2 | 기술분야 | 5% | N/10 | | |
| 3 | 배경기술 | 15% | N/10 | | |
| 4 | 해결과제 | 10% | N/10 | | |
| 5 | 해결수단 | 15% | N/10 | | |
| 6 | 발명의 효과 | 15% | N/10 | | |
| 7 | 도면 설명 | 5% | N/10 | | |
| 8 | 실시예 | 20% | N/10 | | |
| 9 | 청구범위 | 5% | N/10 | | |
| 10 | 요약 | 5% | N/10 | | |
|  | 합계 | 100% | | | 종합 점수 |
```

각 항목에서 감점 사유와 개선안을 함께 보고.

### Step 8. 우선순위별 수정안 제시

위반 항목을 다음 3 레벨로 분류:

- **★★★ 필수 수정 (출원 전 반드시)**: 금지어 0건 위반, 청구항 누락, 부호 매핑 누락
- **★★ 권장 수정 (출원 후 보정 어려움)**: 청구항 수치 도면 노출, 효과 3단 인과 부재
- **★ 미세 조정 (선택)**: 표현 통일, 스토리텔링 매끄러움

각 레벨별로 사용자가 선택적으로 적용할 수 있도록 제안.

## 핵심 점검 노하우 (사용 시 반드시 체크)

이 부분은 실무에서 가장 자주 놓치는 사항이다. 점검 시 우선적으로 본다.

### 1. 청구항 수치 도면 노출 (★★ 위험)

청구항에 "10 mV ≤ V ≤ 4.9 V"가 한정으로 있다면, 도면 박스 라벨에 그 수치를 그대로 노출하지 말 것. 노출 시:
- 권리범위가 도면 수치로 좁게 해석될 가능성
- 보정 시 도면 수정이 별도 절차 필요 (사후 부담)

**대안**: 도면 박스에는 `V_LOW ≤ V ≤ V_HIGH ?` 또는 `유효 범위 내?` 같은 일반화된 표기.

### 2. 동일 객체에 두 부호 사용

본문에서 "TIA"와 "121"을 동의어처럼 혼용하면 권리범위 해석 모호. 한 객체 = 한 부호 + 명칭 원칙.

올바름: "트랜스임피던스 증폭기(이하 'TIA', 121)는 ~한다. 상기 TIA(121)는 ~할 수 있다."

### 3. 청구항 종속 관계와 실시예 종속 관계 불일치

청구항 3이 청구항 1을 종속하는데, 실시예에서 청구항 3의 특징을 청구항 2의 특징과 묶어 설명하면 종속 관계 해석 혼란.

### 4. 도면에만 있는 부호 (★)

명세서에서 한 번도 언급되지 않은 부호가 도면에 보이면 형식 결함. 부호의 설명에는 도면 등장 부호 전수 포함.

### 5. 부호의 설명에만 있는 부호 (★)

본문에 한 번도 안 나오는 부호가 부호의 설명에 있으면, "이 발명에서 왜 이 부호가 필요한가?" 의문 발생. 본문에 최소 1회 등장 필수.

## 컴파운딩 루프 (개선 메모)

점검 후 사용자/검토자/심사관이 추가로 지적한 항목 패턴은 위 마스터 리스트에 누적.

### 누적 학습 항목
- (예시) 회로 발명에서 V_REF 노드의 정전위 조건이 명세서·도면·청구항 3자에 모두 명시되었는지 별도 점검
- (예시) 알고리즘 발명에서 윈도우 크기/오버랩이 청구항·실시예·도면 흐름도에 모두 일치하는지 점검
