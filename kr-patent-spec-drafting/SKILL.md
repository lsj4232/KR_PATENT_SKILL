# 한국 특허 명세서 본문 작성

## 무엇을 하는가

청구항·도면·발명 설명서를 입력받아, 한국 특허청 양식에 따라 명세서 본문 전 섹션을 작성한다. 한국 특허 실무에서 자주 발생하는 가이드라인 위반(금지 표현, 효과 시사, 수단 노출, 청구항 수치 도면 노출 등)을 사전에 회피한다.

**스코프**: 명세서 본문 텍스트 작성에 집중. docx 출력은 `kr-patent-docx-builder` 스킬과 조합. 도면 부호 체계가 미확정이면 먼저 `kr-patent-symbol-design` 스킬로 부호를 확정한 뒤 시작.

## 작업 전 확인 (사용자에게 묻기)

명세서 작성은 권리범위에 직접 영향을 주므로, 다음 항목들은 명시적 확인 없이 임의로 정하지 않는다:

1. **청구항 수정 권한** — "청구항은 그대로 두고 본문만 작성하는가, 청구항도 손볼 수 있는가?" 대부분의 경우 청구항은 고정.
2. **발명의 명칭** — 가능한 한 권리범위를 좁히지 않는 일반적 명칭. 사용자 의도 확인.
3. **부호 체계** — 이미 확정되어 있는지, 신규 설계가 필요한지.
4. **도면 갯수와 구성** — 도면 매핑이 본문 구성을 결정.
5. **참고할 양식·템플릿** — 특허법인마다 표제 양식이 약간 다름.

## 섹션별 작성 가이드

상세 규칙은 `references/ko-patent-style-rules.md`에 정리. 본 SKILL.md에는 핵심만 둔다. 새로 명세서를 쓸 때마다 references 파일을 읽어 패턴을 갱신한다.

### 【발명의 명칭】
- 가능한 한 일반적 명칭. 권리범위 한정 표현 회피.
- "X 및 이를 포함하는 시스템" 같은 패턴으로 카테고리 묶기.

### 【기술분야】
- 1문장. 형식: "본 발명은 ~에 관한 것이다."
- 구성·효과 미언급. 분야만.

### 【발명의 배경이 되는 기술】 (배경기술)
- **금지어**: "종래", "종래 기술" 일체 사용 금지. "기존", "이미", "통상적으로" 등 우회 표현 사용.
- 본 발명의 해결 메커니즘을 노출하지 않는다. 문제 의식만.
- 청구항의 한정 수치(예: 100 dB, 10 mV)를 그대로 노출하지 않는다. 정성적 표현으로 일반화. ("광범위한 동적 영역" 등)
- 마무리: "이에, ~이 요구되고 있는 실정이다." 같은 자연스러운 브릿지로 【해결과제】 연결.

### 【해결하고자 하는 과제】 (해결과제)
- **목적형으로만 작성.** "~을 이용하여(수단) ~하는 것이다" 형식 금지.
- 해결 수단 시사 금지. 예: "정량적으로 판정하여" → 청구항의 신호 생성부 메커니즘 노출, 위험.
- 효과 시사도 부분적으로 위험하지만 목적과 효과는 일부 겹쳐도 무방. 명확한 효과 표현("~을 확보할 수 있도록")보다는 문제의식("~이 손실되는 현상을 저감하기 위한")이 안전.
- 5-8개 목적을 번호 매김 또는 첫째/둘째/셋째 형식으로.

### 【과제의 해결 수단】 (해결수단)
- **청구항 1 (독립항) 통합 서술 → 종속항 2, 3, … 순서대로** "본 발명과 관련된 일 예에 따르면, ~할 수 있다" 패턴.
- 독립항 부분은 "~한다", 종속항 부분은 "~할 수 있다"로 어미 구분 (가능 vs 필수의 권리해석상 신호).
- 각 구성군 종료 시 "~함으로써, ~의 효과를 얻을 수 있다"로 효과 즉시 기재.
- 청구항 누락 0건. 모든 청구항이 본 섹션에 풀려서 등장해야 함.

### 【발명의 효과】 (효과)
- **3단계 인과 구조**: 구조 → 작동 메커니즘 → 사용자 이점
  - 예: "다중 게인 병렬 증폭부를 채널별로 배치함으로써(구조), 코어와 헤일로 영역의 빔 전류가 동일한 측정 절차에서 각자 적합한 감도로 처리되어(메커니즘), 별도의 재측정 없이 광범위한 동적 영역의 빔 분포가 단일 절차로 획득되는 효과를 얻을 수 있다(이점)."
- 5-8개 효과 + 마무리 종합 1-2문장
- 해결과제와 1:1 매핑 추천 (해결과제 N번 → 효과 N번)
- 종속항이 추가한 부가 효과도 별도 항목으로

### 【도면의 간단한 설명】
- 형식: "도 N은 본 발명의 일 실시예에 따른 X를 보인 (도면 종류)이다."
- "보인"(또는 "나타낸") 표현 통일
- 도면 종류: 사시도/평면도/단면도/회로도/블록도/순서도/그래프 등

### 【발명을 실시하기 위한 구체적인 내용】 (실시예)
- 도입부: "본 발명의 이점 및 특징, 그리고 그것들을 달성하는 방법은 첨부되는 도면과 함께 상세하게 후술되어 있는 실시예들을 참조하면 명확해질 것이다. 그러나 본 발명은 이하에서 개시되는 실시예들에 한정되는 것이 아니라 서로 다른 다양한 형태로 구현될 수 있으며, …" (표준 해석 규정 문구)
- **도 1 → 도 N 순서**대로 각 도면을 상세 설명
- 각 도면 설명에 6관점 적용: 구조 / 기능 / 관계 / 메커니즘 / 효과 / 변형례
- **"왜 이런 구조인가"** 표현을 자주 사용: "기술적 이유는 ~을 위한 것이다."
- **사전 패턴**으로 핵심 용어 정의: "여기에서, X라 함은 ~를 의미할 수 있다."
- **동의어 방어망**: "X는 Y로도 명명될 수 있다." (선행기술이 다른 용어를 쓸 가능성에 대비)
- 청구항 14-20개를 모두 본 섹션에 반영. 청구항 누락 0건.
- 변형례: "본 발명의 일 변형예에 따르면 ~할 수도 있다." 다양한 변형 가능성 명시 (권리범위 확장)

### 【부호의 설명】
- 부호 체계대로 한 줄에 하나씩

### 【청구범위】
- 사용자가 제공한 청구항 원문 그대로. 절대 임의 변경 금지. 한 글자도 손대지 않음.

### 【요약】 (요약서)
- 청구항 1을 그대로 복사하지 않음. 의역.
- 효과 1-2문장 포함.
- 형식: "본 발명은, ~ 시스템에 관한 것으로, ~을 포함한다. 이를 통해 ~한 효과를 얻을 수 있다."

### 【대표도】
- 사용자 지정. 통상 도 1.

## 한국 특허 금지 표현 / 위험 표현 일람

| 표현 | 위험도 | 대안 |
|---|---|---|
| "종래" / "종래 기술" | ★★★ 금지 | "기존", "이미", "통상적으로", 우회 표현 |
| "구성되는" | ★★★ 금지 (개방형이 closed로 해석될 우려) | "이루어지는", "포함하는" |
| "필수적이다" | ★★ 위험 | "바람직하다", "할 수 있다" |
| "본 발명의 특징은 ~이다" (정의 시도) | ★★ 위험 (Estoppel 우려) | "본 발명의 일 측면에 따르면" |
| "최적의" / "최고의" / "최상의" | ★ 주의 | "적절한", "바람직한" |
| 인체 위해/유해 표현 | ★★★ 금지 | 의도 우회 |
| 폭발/위험 표현 | ★★ 위험 (안전성 의문) | 약화된 표현 |
| 청구항 한정 수치 그대로 배경기술에 노출 | ★★ 위험 | 정성적 표현 |
| 효과 표현을 해결과제에 노출 | ★ 주의 | 문제의식만 |

## 한국 특허 권장 표현

| 상황 | 표현 |
|---|---|
| 독립항 풀어쓰기 어미 | "~한다." |
| 종속항 풀어쓰기 어미 | "~할 수 있다." |
| 변형예 도입 | "본 발명의 일 변형예에 따르면" |
| 다른 실시예 도입 | "본 발명의 다른 실시예에 따르면" |
| 효과 즉시 기재 | "이러한 구성에 의해, ~효과를 얻을 수 있다." |
| 용어 정의 | "여기에서, X라 함은 ~를 의미할 수 있다." |
| 동의어 방어 | "X는 Y로도 명명될 수 있다." |
| 기술적 이유 설명 | "기술적 이유는 ~을 위한 것이다." |

## 스토리텔링 일관성 (★중요)

배경기술 → 해결과제 → 해결수단 → 효과는 **하나의 스토리**로 흘러야 한다. 각 섹션은 다음 섹션을 자연스럽게 유도한다.

| 섹션 | 다음 섹션으로의 브릿지 |
|---|---|
| 배경 | "~이 요구되고 있는 실정이다." → 해결과제 도입 |
| 과제 | "이러한 목적을 달성하기 위해, …" → 해결수단 도입 |
| 수단 | "~함으로써 ~효과를 얻을 수 있다." → 효과 도입 |
| 효과 | 청구항 누락 없이 모두 다룸 → 실시예에서 상세 |

스토리 일관성 점검: 배경에서 제기한 문제가 효과 섹션에서 모두 해소되는가?

## 작업 순서

1. **자료 확인**: 청구항, 도면, 부호 체계, 기술내용설명서 모두 준비됐는지 점검. 부호 체계 미확정 → `kr-patent-symbol-design`으로 먼저.
2. **사용자에게 작업 전 확인 사항 질문** (위 "작업 전 확인" 섹션의 5개 항목)
3. **스토리 설계**: 배경→과제→수단→효과 큰 흐름 1페이지로 정리, 사용자 OK 받기
4. **섹션 순서대로 작성**: 발명의 명칭 → 기술분야 → 배경 → 과제 → 수단 → 효과 → 도면 설명 → 실시예 → 부호 설명 → 청구범위 → 요약 → 대표도
5. **금지어 0건 확인**: 작성 후 `grep`으로 "종래", "구성되는" 검색
6. **청구항 매핑 확인**: 모든 청구항이 해결수단·실시예에 반영됐는지
7. **출력**: 텍스트 형태로 사용자에게 보여주거나, `kr-patent-docx-builder` 스킬로 docx 생성

## 출력 후 자가 점검 체크리스트

작성 완료 후 다음을 모두 확인. 하나라도 미달이면 수정:

- [ ] 배경기술에 "종래" 단어 0건
- [ ] 본문 전체에 "구성되는" 0건
- [ ] 해결과제가 목적형이며 수단 시사 표현 없음
- [ ] 청구항 수치(임계값 등)가 배경기술에 노출되지 않음
- [ ] 효과 각 항목이 3단 인과(구조→메커니즘→이점) 구조
- [ ] 청구항 N개 모두 해결수단에 풀어쓰기 등장 (누락 0)
- [ ] 청구항 N개 모두 실시예에 등장
- [ ] 부호의 설명에 등재된 모든 부호가 본문에 최소 1회 등장
- [ ] 본문에 등장하는 모든 부호가 부호의 설명에 등재
- [ ] 청구항 원문 한 글자도 변경 없음
- [ ] 요약이 청구항 1 복붙이 아님

## 컴파운딩 루프 (개선 메모)

명세서 작성 후 검토자(다른 변리사, 발명자, 심사관)로부터 피드백받은 패턴은 references/ko-patent-style-rules.md에 추가 누적.

### 누적 학습 항목
- (예시) 회로 발명에서 V_REF = V_DRIVE 같은 "전압 정합 조건"은 시스템 레벨 효과로 청구항·실시예 모두에 명시할 것
- (예시) 알고리즘 발명에서 윈도우 크기/오버랩/업데이트 주기를 변형예에서 명시
